## 3. 技术栈选型分析

### 3.1 通信层

#### 选型：HTTP/gRPC 双协议

| 协议 | 适用场景 | 优势 | 劣势 |
|------|---------|------|------|
| **HTTP/JSON** | 外部 API（BFF） | 通用性强，易调试 | 体积大，无类型 |
| **gRPC** | 服务间通信 | 高性能，protobuf 类型安全，流式支持 | 调试复杂，需要代理 |

**决策**：BFF 层用 HTTP/JSON（便于前端接入和调试），内部服务间用 gRPC。

**参考极客时间**：webook 项目 BFF 层统一用 HTTP，gRPC 用于 article-server、bff-server 等微服务通信。

---

#### 选型：Protocol Buffers

**理由**：
1. 代码生成强制类型约束，减少运行时错误
2. JSON 可读性好，便于调试
3. grpc + protobuf 是 Go 微服务事实标准

```protobuf
// 示例：对话请求
message ChatRequest {
  string conversation_id = 1;
  string content = 2;
  V5Features features = 3;
  string model = 4;
}

message ChatResponse {
  string content = 1;
  string thinking = 2;      // 思考过程
  V5Features features = 3; // 响应特征
  int32 tokens_used = 4;
}
```

### 3.2 Web 框架

#### 选型：Gin

| 维度 | Gin | Kratos | go-zero |
|------|-----|--------|---------|
| 性能 | ★★★★★ | ★★★★ | ★★★★ |
| 生态 | ★★★★ | ★★★★ | ★★★★★ |
| 上手难度 | ★ | ★★★ | ★★★ |
| 内嵌功能 | 少 | 中 | 多（全） |
| 适用场景 | 轻量 BFF | 完整微服务 | 业务型微服务 |

**决策**：BFF 层用 Gin（轻量、灵活），参考极客时间 gin/ 目录的用法。

**理由**：
- Gin 性能优秀（基于 httprouter）
- 中间件生态丰富（CORS、Logger、Recovery）
- 路由语法简洁
- 代码量少，维护成本低

**替代考虑**：如果后续业务复杂度提升，可迁移到 go-zero 或 Kratos。

---

### 3.3 数据库

#### 选型：MySQL (GORM) + Redis

| 数据类型 | 存储 | 理由 |
|---------|------|------|
| 用户、会话、消息 | MySQL | 关系型，事务支持，强一致性 |
| 会话缓存 | Redis | 高频读写，热数据 |
| 令牌/配额 | Redis | 原子操作，高性能 |
| 记忆向量 | 延迟决定 | v1 用文本搜索，v2 考虑向量库 |

**MySQL 选型理由**：
- 成熟稳定，运维成本低
- GORM 简化 CRUD
- 极客时间项目大量使用，有参考
- 事务支持（用户余额、消息计数）

**Redis 选型理由**：
- 会话上下文（10-50 条消息）频繁读写
- 分布式锁（对话并发控制）
- 限流计数器
- pub/sub（实时推送）

**替代考虑**：

| 替代方案 | 适用场景 | 迁移成本 |
|---------|---------|---------|
| PostgreSQL | 复杂查询 | 低（GORM 切换驱动）|
| MongoDB | 消息无 schema | 中（需重写 DAO）|
| TiDB | 超大表 | 高（运维差异）|

### 3.4 消息队列

#### 选型：Kafka (Sarama)

**事件驱动架构需求**：
```
┌──────────┐    事件     ┌──────────┐
│ Chat Svc │ ──────────→│ Analysis │
└──────────┘             └──────────┘
      │                        │
      │ 事件                    │ 事件
      ▼                        ▼
┌──────────┐             ┌──────────┐
│ Memory   │             │ 通知服务  │
│ Service  │             │ (未来)   │
└──────────┘             └──────────┘
```

**为什么需要 Kafka**：
1. **解耦**：Chat Service 只需发事件，不知道谁消费
2. **异步**：LLM 响应后可立即返回，特征分析异步处理
3. **削峰**：高峰期消息积压，平滑处理
4. **可观测**：所有事件可 replay

**选型理由（vs 其他 MQ）**：

| 对比 | Kafka | RabbitMQ | RocketMQ |
|------|-------|----------|----------|
| 吞吐量 | ★★★★★ | ★★★ | ★★★★ |
| 延迟 | 低 | 低 | 低 |
| 生态 | ★★★★★ | ★★★ | ★★★ |
| 极客时间使用 | ✓ | - | - |
| 运维复杂度 | 中 | 低 | 中 |

**Sarama**：Go 语言 Kafka 客户端，极客时间 sarama/ 模块有使用示例。

**延迟考虑**：如果未来 v1 对延迟要求极高（< 100ms），可考虑：
- 同步处理（Chat Service 直接调用 Analysis）
- Redis Streams 替代 Kafka（更轻量）

### 3.5 链路追踪

#### 选型：OpenTelemetry + Zipkin

```
请求链路：
User → BFF → Chat Service → LLM Provider
         ↓          ↓
    [Zipkin]   [Zipkin + Trace ID]
```

**选型理由**：
- 极客时间 opentelemetry/ 模块有实现
- OpenTelemetry 是 CNCF 标准，跨语言
- Zipkin 轻量，部署简单
- 可无缝迁移到 Jaeger / Tempo

### 3.6 依赖注入

#### 选型：Wire（编译期 DI）

```go
// wire.go
//go:build wireinject

func InitializeChatService(...) (*ChatService, error) {
    wire.Build(
        NewDB,           // 数据库
        NewRedis,        // Redis
        NewLLMFactory,   // LLM
        NewChatService,
    )
    return nil, nil
}
```

**优势**：
1. 编译期检测依赖错误，早发现
2. 无反射，性能好
3. 代码可读性强
4. 极客时间 wire/ 模块有示例

**替代考虑**：
- dig / fx：运行时反射，启动快但运行时错误
- 手动注入：简单场景够用，但扩展麻烦

### 3.8 LLM Provider 框架

#### 自研 Provider 抽象

```go
// 统一接口
type Provider interface {
    Chat(ctx context.Context, req ChatRequest) (*ChatResponse, error)
    Name() string
    SupportsThinking() bool  // 是否支持思考过程
}

// 接入方式
factory.Register("coze", NewCozeProvider(...))
factory.Register("openai", NewOpenAIProvider(...))
factory.Register("claude", NewClaudeProvider(...))
```

**优势**：
- 统一 API，切换模型无感知
- 插件化，动态注册
- 易于测试（Mock Provider）

**Provider 对比**：

| Provider | 思考过程 | API 稳定 | 成本 | 推荐 |
|----------|---------|---------|------|------|
| Coze | ✓ | ✓ | 中 | ⭐ MVP |
| OpenAI | ✓ | ✓ | 高 | ⭐ |
| Claude | ✓ | ✓ | 高 | ⭐ |
| 本地代理 | ? | 视实现 | 低 | 私有部署 |

## 4. 技术决策记录（TDR）

### TDR-001：是否使用微服务？

| 选项 | 决策 | 理由 |
|------|------|------|
| 单体 | ❌ | MVP 后再拆分 |
| **微服务** | ✅ | 参考 webook 架构，预留扩展 |
| Serverless | ❌ | LLM 延迟不可控 |

**结论**：v1 用单体快速迭代，架构预留微服务拆分能力。

---

### TDR-002：是否引入 Kafka？

| 选项 | 决策 | 理由 |
|------|------|------|
| 无 MQ | ✅ v0 | MVP 同步处理足够 |
| **Kafka** | ✅ v1 | 事件驱动，解耦服务 |
| Redis Streams | 备选 | 如果 Kafka 运维成本过高 |

---

### TDR-003：存储选型

| 选项 | 决策 | 理由 |
|------|------|------|
| **MySQL + Redis** | ✅ | 成熟稳定，有极客时间参考 |
| PostgreSQL | 备选 | 如果需要复杂 JSON 查询 |
| MongoDB | 备选 | 如果消息 schema 变化频繁 |

---

### TDR-004：是否支持流式输出？

| 选项 | 决策 | 理由 |
|------|------|------|
| 非流式 | ✅ v0 | 实现简单 |
| **Server-Sent Events** | ✅ v1 | 前端体验好，实现成本中等 |
| WebSocket | v2 | 需要双向通信时考虑 |

## 5. 风险分析与应对

| 风险 | 影响 | 概率 | 应对策略 |
|------|------|------|---------|
| LLM API 不稳定/限流 | 高 | 中 | 多 Provider 降级 + 本地规则兜底 |
| 技术栈过于复杂 | 中 | 低 | MVP 先用 Gin 单体，渐进演进 |
| V5 分析准确率不足 | 中 | 中 | 上线后收集反馈，持续调优 |
| 冷启动慢 | 低 | 低 | 预热 + 缓存优化 |
| 数据安全合规 | 高 | 低 | GDPR/隐私政策，用户数据加密 |
| 云服务商锁定 | 中 | 低 | Docker + K8s 可迁移 |

## 6. 技术演进路线

```
Phase 1: MVP (单体)
─────────────────────────────────────────
Gin HTTP API + MySQL + Redis
单进程，无 Kafka，无 gRPC
目标：跑通核心流程

Phase 2: 扩展 (拆分)
─────────────────────────────────────────
微服务化：
  BFF (Gin) → Chat Service (gRPC)
            → Analysis Service (gRPC)
引入 Kafka 事件总线
目标：高并发支撑

Phase 3: 成熟 (生态)
─────────────────────────────────────────
开放 API + WebSocket
向量记忆库
多租户 SaaS
目标：商业化
```
